# Variant Gate Cross-Review Notes

1. `run_variant_gate.py` (scripts/eval_harness/run_variant_gate.py:229-311) re-runs direct and variant prompts through the raw `OllamaClient` interface with a minimal budget/rules prompt. That bypasses the `QuickThinkEngine` routing/plan-repair path, so deployment metrics may not reflect the real lite/two_pass behavior that operators release. Consider instrumenting the gate to run through the same engine or at least replay the chosen plan/repair prompts to confirm compatibility.
2. The deployment gate doc (docs/evals/deployment_gate_2026.md:1-80) calls for paired quality/statistics plus edge-case passes, which is aligned with the script, but it omits any mention of how pipeline winners were generated or how the edge-case hits are validated. It may be worth linking the pipeline1 variant artifact (`experiments-local/pipeline1-runs/.../selected_variants.json`) and defining the acceptance criteria for the edge-case suite (e.g., zero constraint errors) so future readers know what “approve only if …” concretely means.
3. The new edge-case prompt set (docs/evals/prompt_set_edge_cases.jsonl) already covers constraining tokens, formatting, and multi-turn continuity. We should track coverage (per group) and flag if a variant fails the entire subset rather than just recording lift cases; without that, a variant could pass the gate by only improving a few prompts while still regressing on the rest.
